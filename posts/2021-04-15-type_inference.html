타입 추론 Type inference

Posted on April 15, 2021

하스켈이 가진 꽤 멋진 특징인데, 이게 또 익숙해지기 전까지는 머리를 아프게 합니다. 하스켈은 초반에는 타입 추론과의 전쟁인 것 같습니다. 나중에는 친구가 될지 어쩔지 모르지만, 지금까지는 전쟁중임에 틀림 없습니다.

다른 부품과의 접점을 보고 타입을 결정하진 않아

아래 코드를 보면 GHC가 알아서 추론할 수 있을 것처럼 보입니다.

func :: a -> a
func x = x + 1
-- (+)는 인스턴스가 여러가지 있습니다.

intFunc :: Int -> Int
intFunc x = x + 2

-- func와 intFunc가 "만난다면" 추론이 될 것 같지 않나요?
main :: IO ()
main = do
    print $ func ( intFunc 1)

func에 intFunc 결과값 Int 타입이 넘어가니, a타입은 GHC가 Int로 추론할 것 같은데, 에러가 납니다.

    • No instance for (Num a) arising from a use of ‘+’
 ... 
2 | func x = x + 1

다른 함수와 만나기 전에는 어느 인스턴스의 (+)를 고를지 알 수 없습니다. 항상 적당한 다른 함수와 꼭 만난다고 가정할 수 없습니다. 단독으로도 모호성이 없는 코드여야 합니다. 나중에 조립할 때 타입을 결정할 수 있다면 좀 더 유연하지 않을까요?

클래스는 미리 알려줄게, 구체적인 코드는 나중에 결정해

func :: Num a => a -> a
...

Num 이라는 클래스를 알려주면 GHC는 불만 없이 다음으로 넘어갑니다. 나중에 조립할 때 GHC가 Num 인스턴스가 넘어오는지 살펴 보기로 하고 넘어 갑니다. 저는 이 걸 구체 코드 결정을 “뒤로 미룬다”고 표현합니다.

항상 완벽한 구체 타입을 알아야 하는 건 아니다

func :: [a] -> Int
func xs = length xs

main :: IO ()
main = do
    print $ func [1,2,3]

첫 번째 논리대로 하면, func 정의 단독으로 a에 대한 어떤 힌트가 있어야 합니다. 하지만 에러가 나지 않습니다. 위의 첫 번째 예는 (+)란 함수가 모호해서 나는 에러였습니다. 여기선 모호한 게 없습니다. length 함수가 모호하지 않습니다.
참고 - 클래스 제약 포스트

> :t length
length :: Foldable t => t a -> Int

length 함수는 리스트 구조가 됐든, 트리 구조가 됐든… Foldable 클래스의 인스턴스이기만 하면 받아들입니다. 안에 들어 있는 a타입은 뭔지 모르지만 foldable이 가능한 [] 리스트 구조임은 명시했으니, 전체적으로 모호한게 없는 상태입니다. 아래는 또 다른 예시입니다. 가장 바깥 타입 생성자(여기서는 Maybe)만 명시해 주면 모호하지 않습니다.

isJust         :: Maybe a -> Bool
isJust Nothing = False
isJust _       = True

위 Maybe a에서 a타입은 알 필요가 없습니다. 가장 바깥만 알면 됩니다.
