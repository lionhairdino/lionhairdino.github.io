forall

Posted on May 4, 2021

에러 관련 포스트들에서 몇 번 강조했는데 forall은 말 그대로 모든 타입입니다. 어떤 타입으로든 카멜레온처럼 변할 수 있다가 아니라, 어떤 타입도 처리할 수 있어야 한다는 뜻입니다. ( 참고 - 컴파일 에러 읽기 - rigid type variable ) 그런데, 이 걸 명시적으로 써야 하는 경우가 있습니다. 바로 Rank 개념이 필요한 경우입니다.

RankNTypes 설명은 남현욱님의 설명이 좋습니다. 여기 글은 RankNType 글을 본 후 생각해 볼만한 내용입니다.

위 사이트에서 예를 든 소스를 가져와 약간 변형해 보겠습니다.

{-# LANGUAGE RankNTypes #-} 

-- foo :: (forall a. [a] -> Int) -> [b] -> [c] -> Bool
foo :: (forall a. [a] -> Int) -> forall a. [a] -> forall a. [a] -> Bool
foo f x y = f x == f y

main = do
    print $ foo length [1,2,3,4] ['a','b','c']

foo 함수가 실행될 때, 각 a들은 모두 다른 타입이 될 수 있습니다. forall을 특정 범위에 한정되게 붙이면 a 글자를 b나 c등으로 바꿔도 상관 없습니다. 람다 대수학의 alpha equivalance와 같은 의미입니다. 그런데 아래와 같이 영향을 미치는 범위를 명확히 표시하기 위해 괄호로 감싸주면

foo :: (forall a. [a] -> Int) -> (forall a. [a]) -> (forall a. [a]) -> Bool

에러가 납니다.

    • No instance for (Num a) arising from the literal ‘1’ ------------------(1)
      Possible fix:
        add (Num a) to the context of
          a type expected by the context:
            forall a. [a]
    • In the expression: 1
7 |     print $ foo length [1,2,3,4] ['a','b','c']
  |                         ^

    • Couldn't match expected type ‘a’ with actual type ‘Char’ --------------------(2)
      ‘a’ is a rigid type variable bound by
        a type expected by the context:
          forall a. [a]
    • In the expression: 'a'
7 |     print $ foo length [1,2,3,4] ['a','b','c']
  |                                   ^^^

왜 그럴까요? 일단, 하나는 No instance, 하나는 Couldn't match 입니다.

  • No instance인 이유는, 리터럴 1이 Int나 Integer 같이 딱 하나의 타입으로 추론되지 않고 Num 클래스의 인스턴스란 것 까지만 추론되었기 때문입니다.
  • Couldn't match는 딱 하나의 타입 Char로 추론됐는데 서명에는 모든 타입으로 지정rigid되어 있어, 서로 매치되지 않는다는 에러입니다. 참고 - rigid type variable

여기서부턴 추측입니다.
괄호로 감싸져 있을 땐, 서명에서 영향이 있는 게 아니라, 나중에 들어오는 값이 진짜 forall a. [a] 가 들어와야 합니다. 모든 타입 중에 어느 한 타입이 아니라, 진짜로 모든 타입을 받을 수 있는 무언가가 들어와야 합니다.

  1. No instance : 클래스는 추론됐는데, 해당 타입을 키로 하는 인스턴스는 없다.
  • 서명 체크를 지나, 나중에 코드 조립 중 들어온 타입을 보니 1이 들어왔습니다.
  • 1이란 리터럴로 추론한 타입은 Num 클래스의 인스턴스입니다.
  • 지금 정해지지 않은 타입, 모든 타입 a가 Num의 인스턴스여야 한다는 말입니다. a는 모든 타입인데, Num 인스턴스 중 모든 타입용 인스턴스는 당연히 없습니다.
  1. Couldn't match : 특정 타입까지 추론됐는데, 다른 조건에서 추론된 타입이 일치하지 않는다.
  • 서명 체크를 지나 , 나중에 코드 조립 중 들어온 타입을 보니 'a'가 들어왔습니다.
  • 'a'를 보고 추론할 수 있는 타입은 Char 하나입니다. 구체 타입 Char로 추론되었습니다.
  • 지금 정해지지 않은 타입, 모든 a는 모든 타입이지 아직 하나의 구체 타입이 아닙니다. 모든 타입과 Char는 매치되지 않습니다.

만일 서명에서 영향을 주는 forall a였다면, GHC는 “이 매개 변수에 들어오는 타입은 뭐든 상관없어. 이 함수는 어떤 타입이든 처리할 수 있어” 라고 받아들입니다. (나중에 코드 조립할 때 들어오는 타입은 모든forall a 타입일 필요 없습니다. Int가 들어와도 되고, Char가 들어와도, … 됩니다.)
하지만 괄호가 있으면, GHC는 “나중에 들어오는 타입이 어느 한 타입이 아니라, 모든 타입 forall a이어야만 해”라고 받아들입니다.

foo의 첫 번째 매개 변수의 타입은 forall a. [a] -> Int 가 아니라, (forall a. [a] -> Int)로 괄호로 싸여져 있습니다. 서명 단계가 아니라, 조립 단계 때 forall a타입을 다루는 함수가 들어오는지 확인한다는 말입니다. 이 매개 변수를 통해 들어온 함수의 타입은 어느 한 타입으로 고정된 타입이 아닌 forall a. [a] -> Int 로 고정된 타입입니다. 함수 몸체에서 쓰일 때마다 forall a를 끌고 다니며 독립적으로 체크한다는 말입니다. 그래서 length란 함수가 들어 와서 f x == f y 에서 쓰일 때

  1. x에 적용한 length 함수는 forall a. [a] -> Int 타입으로 서명에서 정한 조건을 통과했으니, 이제 x의 타입에 따라 구체 타입을 추론합니다.
  2. y에 적용한 length 함수도 forall a. [a] -> Int 타입으로 여기서도 통과하고, 구체 타입을 독립적으로 추론합니다.

아직 명확하지 않은데, 하나는 함수 내부와 외부가 일치하는지 보는 서명 체크 단계에 쓰이고, 다른 하나는 함수 내부 코드 조립 단계로 이해하면 되지 않을까 합니다. 명확한 자료가 찾아지면 보충하도록 하겠습니다.

모든 타입 forall a. a란 말은

  1. 서명 단계에선 “어떤 타입으로 결정되어도 돼” - “조립할 때 해당 매개 변수로 어떤 타입이 들어오든 상관 없어”
  2. 내부 조립 단계에선 “모든 타입을 다 처리할 수 있어” -

예를 들면,
서명 단계의 forall a. a란 곳으로 Int가 들어오면 통과할 수 있지만,
조립 단계에서 꺼내 온 (forall a. a) 로는 Int가 통과할 수 없습니다. Int는 모든 타입이 아닙니다.

괄호 안은 lazy하게 해석한다 말하면 될까요? 설명하면서도, 단어 선택이 혼동을 주고 있어, 좀 더 고민해서 바꿔야 할 것 같습니다.
