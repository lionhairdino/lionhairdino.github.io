Haskell, 하스켈, 해스클?

ML 스타일 문법이 너무 예쁘고(처음에는 ML을 몰라 하스켈만의 문법인 줄 알았습니다.) Async 라이브러리가 너무 간결해 빠져들게 된 언어입니다. 현재 도착해 있는 곳은 모나드 -> 모나드 트랜스포머 -> mtl 스타일 -> Type Level programming -> ...즈음 와 있습니다.

여기 글들은

  • 처음 하스켈을 만나는 분들을 위한 글이 아닙니다. 입문서를 반쯤 보고, 다른 소스들을 읽기 시작할 때 보기 적당합니다.
  • 학문적으로 정교한 정의를 익히는 것보다, 개념이 도입된 이유나, 핵심 아이디어에 관심이 더 많습니다.
  • 함수형을 맛보기 위해 하스켈을 만져보는 수준을 넘어, 현업에서 사용할 때 필요한 정보를 담으려 노력했습니다.
  • 검증이 필요한 글들입니다.

하스켈의 현란한 로직, 패턴들을 보기 전에 다음 질문을 곰곰히 생각해 보면 정말 도움이 됩니다.

"함수는 오직 매개 변수로만 소통하는데, 전역 변수를 어떻게 만들지?"

위 질문을 좀 더 도움되는 질문으로 바꾸겠습니다.

"함수는 오직 매개 변수로만 소통하는데, 전역 변수와 같은 결과를 만들어내는 기능을 어떻게 만들지?"

하스켈은 언어 설계상 상태를 유지할 수 없습니다. 전역 변수는 만들지 못합니다. 하지만, 전역 변수와 같은 기능을 하는 걸 만드는 건 다른 문제입니다. 이 질문이 모나드 이해의 출발점입니다. 함수형 프로그래밍은 최대한 상태 개념을 쓰지 않으며 해결하려 노력하지만, 결국 실용 프로그램에서는 상태 개념을 완전히 빼고 갈 순 없습니다. 상태 개념을 쓰지 않는 함수형 로직(함수형의 가장 바탕이 되는 철학 statless, immutable...)들을 훑어보려면 모나드를 익히지 않아도 되지만, 하스켈을 실제 세계realworld로 가져오려면 반드시 모나드를 익혀야 합니다.

같은 목표를 가진 분들의 피드백을 환영합니다.
